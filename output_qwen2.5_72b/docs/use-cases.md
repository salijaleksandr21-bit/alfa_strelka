## Общие принципы безопасности
При разработке данных сценариев учтены следующие требования безопасности:
1.  **Отсутствие PII:** Система не собирает, не хранит и не передает персональные данные пользователей (ФИО, email, номера счетов и т.д.).
2.  **Защищенный канал:** Все взаимодействия с внешним API должны происходить по защищенному протоколу (HTTPS/TLS).
3.  **Валидация ввода:** Все входные данные проверяются на сервере и клиенте для предотвращения инъекций и XSS-атак.
4.  **Принцип минимальных привилегий:** Сервис обращается к API только с необходимыми правами доступа.

## Юз-кейс 1: Выполнение конвертации валюты (Основной поток)

| Параметр | Описание |
| :--- | :--- |
| **ID** | **UC-01** |
| **Название** | Конвертация валюты |
| **Актор** | Пользователь (Гость) |
| **Ссылки на БТ** | [БТ-01], [БТ-02], [БТ-03], [БТ-05], [БТ-07] |
| **Предусловия** | Пользователь открыл веб-страницу конвертера. Соединение с сервером установлено. |
| **Постусловия** | Пользователь видит результат конвертации. Данные не сохраняются в истории. |

### Шаги сценария
1.  Система отображает интерфейс выбора валюты и поле ввода суммы.
2.  Пользователь выбирает **исходную валюту** из списка (например, USD).
    *   *Безопасность:* Список формируется динамически, исключая вредоносные символы.
3.  Пользователь выбирает **целевую валюту** из списка (например, EUR).
    *   *Безопасность:* Список формируется динамически, исключая вредоносные символы.
4.  Пользователь вводит **числовое значение** суммы.
    *   *Безопасность:* Ввод валидируется на клиенте (только цифры и разделитель).
5.  Система отправляет запрос к внешнему API для получения курса.
    *   *Безопасность:* Запрос передается по HTTPS.
6.  Система получает актуальный курс и мгновенно отображает результат конвертации.
7.  Система очищает временные данные из памяти после закрытия вкладки (если применимо).

## Юз-кейс 2: Смена направления конвертации

| Параметр | Описание |
| :--- | :--- |
| **ID** | **UC-02** |
| **Название** | Смена направления конвертации |
| **Актор** | Пользователь (Гость) |
| **Ссылки на БТ** | [БТ-04] |
| **Предусловия** | Пользователь выбрал валюты и ввел сумму (см. UC-01). |
| **Постусловия** | Валюты поменяны местами, результат пересчитан. |

### Шаги сценария
1.  Пользователь нажимает кнопку **«⇄»** (Сменить валюты).
2.  Система меняет местами значения полей «Исходная валюта» и «Целевая валюта».
3.  Система инициирует пересчет суммы с использованием актуального курса.
4.  Система отображает новый результат конвертации.
    *   *Безопасность:* Убедиться, что при смене не происходит утечки предыдущего состояния через URL или логи.

## Юз-кейс 3: Обработка недоступности API (Кеширование)

| Параметр | Описание |
| :--- | :--- |
| **ID** | **UC-03** |
| **Название** | Обработка ошибки внешнего API |
| **Актор** | Пользователь (Гость) |
| **Ссылки на БТ** | [БТ-06] |
| **Предусловия** | Внешний API недоступен или отвечает с ошибкой. |
| **Постусловия** | Пользователь видит результат на основе кэшированных данных с предупреждением. |

### Шаги сценария
1.  Пользователь пытается выполнить конвертацию (см. UC-01).
2.  Система не получает ответ от внешнего API в течение заданного таймаута.
3.  Система проверяет наличие локального кэша с последним успешным курсом.
4.  Если кэш есть, система отображает результат конвертации.
5.  Рядом с курсом отображается пометка: **«Курс от [дата]. Актуальные данные временно недоступны»**.
    *   *Безопасность:* Дата в кэше должна быть проверена на валидность, чтобы избежать использования устаревших данных, которые могут быть скомпрометированы.
6.  Если кэша нет, система отображает сообщение об ошибке соединения.

## Юз-кейс 4: Обработка некорректного ввода

| Параметр | Описание |
| :--- | :--- |
| **ID** | **UC-04** |
| **Название** | Валидация входных данных |
| **Актор** | Пользователь (Гость) |
| **Ссылки на БТ** | [БТ-03] (через БП 5a) |
| **Предусловия** | Пользователь находится на странице конвертера. |
| **Постусловия** | Ошибка отображена, конвертация не выполнена. |

### Шаги сценария
1.  Пользователь вводит в поле суммы **некорректное значение** (отрицательное число, текст, спецсимволы).
2.  Система перехватывает ввод и проверяет его валидность.
3.  Система **не выполняет** запрос к API.
4.  Система отображает сообщение об ошибке: **«Введите корректную сумму»**.
    *   *Безопасность:* Сообщение об ошибке не должно раскрывать внутреннюю логику системы (например, не должно писать «Ошибка SQL» или «Ошибка валидации JSON»).
5.  Поле ввода очищается или блокируется до ввода корректных данных.

## Примечания по безопасности (Security Notes)

1.  **Отсутствие аутентификации:** В текущих требованиях ([БТ-01]...[БТ-07]) не предусмотрена авторизация. Если в будущем потребуется учетная запись, необходимо реализовать безопасную аутентификацию (OAuth2, MFA) и не хранить пароли в открытом виде.
2.  **Защита от XSS:** Поле ввода суммы должно быть экранировано при выводе на страницу, чтобы предотвратить выполнение вредоносного JavaScript кода.
3.  **Защита от CSRF:** Если в будущем будут добавлены формы отправки данных, необходимо использовать CSRF-токены.
4.  **Логирование:** Логи должны содержать только технические метаданные (IP-адрес, время, статус), но **никогда** не должны содержать введенные пользователем суммы или выбранные валюты в контексте, который можно связать с личностью.

## Заключение
Данные юз-кейсы соответствуют бизнес-требованиям и обеспечивают базовый уровень безопасности для системы конвертации валют. Все сценарии исключают сбор персональных данных и предполагают работу в защищенном окружении.
