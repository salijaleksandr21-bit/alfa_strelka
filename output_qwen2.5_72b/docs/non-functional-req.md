# Нефункциональные требования (НФТ)

Ниже представлены нефункциональные требования, сформированные на основе бизнес-требований и бизнес-процесса, с учетом строгих правил безопасности и этики.

| ID | Категория | Требование |
|----|-----------|------------|
| **НФТ-01** | **Безопасность** | Все пользовательские вводы (включая сумму конвертации и выбор валют) должны проходить строгую валидацию и санитизацию на стороне клиента и сервера для предотвращения атак типа XSS (Cross-Site Scripting) и инъекций. |
| **НФТ-02** | **Конфиденциальность** | Любые данные, сохраняемые в локальном хранилище браузера (например, кэшированные курсы валют согласно БТ-06), должны быть зашифрованы. Личные данные пользователя не должны покидать браузер без явного согласия и необходимости для работы сервиса. |
| **НФТ-03** | **Безопасные настройки по умолчанию** | Система не должна раскрывать внутренние API ключи или учетные данные в клиентском коде (JavaScript). Обращение к внешнему API (БТ-02) должно осуществляться через защищенный прокси-сервер или серверную функцию, чтобы ключи не были доступны для просмотра в исходном коде. |

---

### Примечание по безопасности и функционалу

В бизнес-требовании **БТ-02** («Система должна отображать актуальный курс обмена, полученный из внешнего API») подразумевается интеграция с внешним сервисом.

*   **Риск:** Если реализация будет выполнена прямым вызовом API из клиентского кода (браузера), ключи доступа к API станут публичными, что нарушит принцип **НФТ-03**.
*   **Решение:** В реализации необходимо использовать серверный прокси (Backend-for-Frontend). Клиент отправляет запрос на ваш сервер, сервер запрашивает данные у внешнего API, используя скрытый ключ, и возвращает результат клиенту. Это устраняет риск утечки учетных данных.
