# Нефункциональные требования (НФТ) к системе конвертации валют

Ниже представлены нефункциональные требования, сформированные на основе бизнес-требований (БТ), бизнес-процесса (БП) и принципов безопасности.

| ID | Категория | Требование |
|----|-----------|------------|
| **NFR-SEC-01** | **Безопасность** | Все пользовательские вводы (поле суммы, выбор валют) должны проходить строгую валидацию и санитизацию на стороне клиента и сервера (при наличии) для предотвращения XSS-атак, инъекций и обработки некорректных данных (согласно БТ-03, БП-5a). |
| **NFR-CONF-01** | **Конфиденциальность** | Личные данные пользователя (история операций, предпочтения) не покидают браузер пользователя; если для хранения настроек используется локальное хранилище (localStorage/IndexedDB), оно должно быть зашифровано или изолировано от доступа сторонних скриптов. |
| **NFR-SEC-02** | **Безопасные настройки** | Система не должна раскрывать внутренние API-ключи, токены или учётные данные в клиентском коде (JavaScript); взаимодействие с внешним API (БТ-02) должно осуществляться через защищённый прокси-сервер или серверные вызовы. |
| **NFR-PERF-01** | **Производительность** | Время отображения результата конвертации после ввода суммы не должно превышать 200 мс в нормальном режиме работы (согласно БТ-03 «мгновенно показывает результат»). |

---

### Примечания по безопасности и этике

1.  **Соответствие правилам:**
    *   Включено требование безопасности (NFR-SEC-01) для защиты от инъекций и XSS.
    *   Включено требование о конфиденциальности (NFR-CONF-01) для защиты данных пользователя.
    *   Включено требование о безопасных настройках по умолчанию (NFR-SEC-02) для предотвращения утечки ключей API.
2.  **Анализ функций:** В предоставленных бизнес-требованиях и процессе не обнаружено явных небезопасных предложений (например, отправка логов на внешний сервер без согласия). Однако требования NFR-SEC-02 и NFR-CONF-01 гарантируют, что даже при реализации опциональных функций (например, кэширование БТ-06) безопасность не будет нарушена.
3.  **Запрещённые практики:** В требованиях не предусмотрены хранение паролей в открытом виде, использование `eval()` для пользовательского ввода или передача чувствительных данных без шифрования.
