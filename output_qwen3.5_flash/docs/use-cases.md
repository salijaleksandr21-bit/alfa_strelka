# Юз-кейсы для системы конвертации валюты (Сценарии использования)

**Статус документа:** Черновик  
**Роль:** Аналитик по безопасности  
**Цель:** Описание сценариев использования с учетом требований безопасности, этики и валидации данных.  
**Примечание:** Все сценарии разработаны с принципом минимальных привилегий и отсутствием сбора персональных данных (PII).

---

## UC-01: Открытие страницы конвертера и получение курсов

| Параметр | Описание |
| :--- | :--- |
| **ID** | UC-01 |
| **Название** | Просмотр страницы конвертера и инициализация данных |
| **Актор** | Пользователь (Гость) |
| **Цель** | Получить доступ к интерфейсу конвертации и загрузить актуальные курсы валют |
| **Предусловия** | Пользователь имеет доступ к интернету и современный веб-браузер. |
| **Постусловия** | Страница загружена, курсы валют отображены (или кэш), пользователь готов к вводу данных. |

### Основной поток
1. Пользователь открывает URL веб-приложения в браузере.
2. Система загружает интерфейс согласно [БТ-07].
3. Система инициирует запрос к внешнему API для получения курсов валют.
4. Система сохраняет полученные данные в кэш для последующего использования.
5. Система отображает список доступных валют для выбора.

### Альтернативный поток — API недоступен
- 2a. Если внешний API недоступен, система загружает данные из локального кэша.
- 2b. Система отображает уведомление: «Курс от [дата]. Актуальные данные временно недоступны» (согласно [БТ-06]).

### Заметки по безопасности
- **[БЕЗОПАСНОСТЬ] Отсутствие PII:** Система не запрашивает и не сохраняет данные, идентифицирующие пользователя (email, IP-адрес, cookies аналитики).
- **[БЕЗОПАСНОСТЬ] Защищенный канал:** Все коммуникации с внешним API и клиентом должны осуществляться по протоколу HTTPS для защиты от перехвата данных.
- **[БЕЗОПАСНОСТЬ] Валидация ответа:** Система должна проверять целостность данных, полученных от API, перед отображением.

## UC-02: Конвертация валюты

| Параметр | Описание |
| :--- | :--- |
| **ID** | UC-02 |
| **Название** | Расчет суммы конвертации |
| **Актор** | Пользователь (Гость) |
| **Цель** | Получить результат конвертации выбранной суммы из одной валюты в другую |
| **Предусловия** | Страница открыта, курсы валют загружены (см. UC-01). |
| **Постусловия** | На экране отображается рассчитанная сумма и текущий курс. |

### Основной поток
1. Пользователь выбирает исходную валюту из выпадающего списка (минимум 10 вариантов согласно [БТ-05]).
2. Пользователь выбирает целевую валюту из выпадающего списка (согласно [БТ-01]).
3. Пользователь вводит числовое значение суммы в поле ввода.
4. Система валидирует введенное значение (проверка на тип данных и диапазон).
5. Система мгновенно рассчитывает результат на основе текущего курса ([БТ-02]).
6. Система отображает результат конвертации и примененный курс ([БТ-03]).

### Альтернативный поток — некорректная сумма
- 4a. Если пользователь вводит отрицательное число, нечисловое значение или значение, превышающее допустимый лимит:
    - Система отклоняет ввод.
    - Отображается сообщение: «Введите корректную сумму».
    - Конвертация не выполняется.
    - *(См. Альтернативный поток 5a из Бизнес-процесса)*.

### Заметки по безопасности
- **[БЕЗОПАСНОСТЬ] Валидация ввода:** Поле ввода суммы должно иметь строгую валидацию на стороне клиента и сервера для предотвращения инъекций (XSS, SQLi) и переполнения буфера.
- **[БЕЗОПАСНОСТЬ] Ограничение ввода:** Необходимо установить максимальный лимит на вводимую сумму для предотвращения атак типа DoS (отказ в обслуживании) через генерацию тяжелых вычислений.
- **[БЕЗОПАСНОСТЬ] Минимальные привилегии:** Пользователь не имеет прав на изменение курсов валют или конфигурации системы.

## UC-03: Смена направления конвертации

| Параметр | Описание |
| :--- | :--- |
| **ID** | UC-03 |
| **Название** | Обмен местами исходной и целевой валюты |
| **Актор** | Пользователь (Гость) |
| **Цель** | Быстро инвертировать направление конвертации без ручного выбора валют |
| **Предусловия** | Пользователь выбрал исходную и целевую валюты. |
| **Постусловия** | Валюты поменяны местами, результат пересчитан. |

### Основной поток
1. Пользователь нажимает кнопку «⇄» (Сменить валюты).
2. Система меняет местами значения полей «Исходная валюта» и «Целевая валюта».
3. Система пересчитывает результат конвертации на основе новых параметров.
4. Система обновляет отображение результата ([БТ-04]).

### Заметки по безопасности
- **[БЕЗОПАСНОСТЬ] Целостность данных:** При смене валют система должна убедиться, что выбранные валюты остаются в допустимом списке ([БТ-05]), чтобы избежать ошибок логики.
- **[БЕЗОПАСНОСТЬ] CSRF:** Если бы действие изменяло состояние (например, сохраняло историю), требовалась бы защита от CSRF. В данном случае (только расчет) риск минимален, но интерфейс должен быть защищен от подделки запросов.

## UC-04: Работа с кэшированными данными

| Параметр | Описание |
| :--- | :--- |
| **ID** | UC-04 |
| **Название** | Использование закешированного курса при сбое API |
| **Актор** | Пользователь (Гость) |
| **Цель** | Продолжить работу конвертера при временной недоступности внешнего источника данных |
| **Предусловия** | Внешний API недоступен или отвечает с ошибкой. |
| **Постусловия** | Пользователь видит результат конвертации с пометкой о статусе данных. |

### Основной поток
1. Система пытается получить актуальный курс из внешнего API.
2. Получена ошибка соединения или тайм-аут.
3. Система обращается к локальному кэшу с последними успешными данными.
4. Система отображает результат конвертации.
5. Рядом с курсом отображается пометка: «Курс от [дата]. Актуальные данные временно недоступны» ([БТ-06]).

### Заметки по безопасности
- **[БЕЗОПАСНОСТЬ] Прозрачность:** Пользователь должен быть явно уведомлен о том, что данные не являются актуальными в реальном времени, чтобы избежать финансовых потерь из-за устаревшей информации.
- **[БЕЗОПАСНОСТЬ] Защита кэша:** Кэш должен храниться в безопасном хранилище, доступном только для чтения для текущего процесса конвертации.

## Общие рекомендации по архитектуре безопасности

1.  **Отсутствие аутентификации:** Поскольку функционал не требует хранения данных пользователя, реализация системы аутентификации не требуется, что снижает поверхность атаки.
2.  **Обезличивание:** Даже если в будущем потребуется логирование ошибок, логи должны содержать только технические данные (IP, User-Agent), без привязки к личности пользователя.
3.  **Обновление зависимостей:** Внешние библиотеки для работы с API и UI должны регулярно обновляться для устранения уязвимостей.
4.  **Запрет на выполнение кода:** Интерфейс не должен поддерживать выполнение пользовательского кода (например, через `eval` или динамические скрипты).
