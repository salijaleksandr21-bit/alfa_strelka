Как инженер по требованиям, специализирующийся на безопасности, я проанализировал предоставленные бизнес-требования, юз-кейсы, нефункциональные требования и Features. На их основе я разработал детальные функциональные требования (ФТ), каждое из которых проверено на соответствие правилам безопасности и этики.

### Функциональные требования (ФТ)

#### ФТ-01: Создание новой задачи

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-01 |
| **Источник** | БТ-01, БТ-02, БТ-07, UC-01, Features (поле «Описание» необязательное) |
| **Описание** | Система должна предоставлять пользователю возможность создать новую задачу, указав её название (обязательно) и описание (опционально). После создания задача получает статус «К выполнению». |
| **Входные данные** | Пользовательский ввод: `название` (строка, обязательное), `описание` (строка, опциональное). |
| **Ожидаемый результат** | 1. Система отображает форму с полями «Название» и «Описание» после нажатия кнопки «Новая задача». <br>2. Система валидирует поле «Название»: если оно пустое или состоит только из пробелов, отображается сообщение «Название задачи обязательно», задача не создаётся. <br>3. Система создаёт задачу с уникальным ID, статусом «К выполнению» и сохраняет её в локальном хранилище. <br>4. Новая задача отображается на доске в колонке «К выполнению». |
| **Примечания по безопасности** | **Валидация и санитизация ввода:** Все строковые поля (`название`, `описание`) должны быть проверены на длину (максимум 500 символов для названия, 2000 для описания) и очищены от потенциально опасных HTML-тегов и JavaScript-кода (например, с помощью DOMPurify или эквивалентной библиотеки) перед сохранением и отображением. <br>**Кодирование вывода:** При отображении названия и описания задачи на доске, данные должны быть закодированы для HTML (например, с использованием `textContent` вместо `innerHTML`), чтобы предотвратить XSS-атаки. |

#### ФТ-02: Изменение статуса задачи

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-02 |
| **Источник** | БТ-02, БТ-03, БТ-07, UC-02, Features (цветные метки статусов, анимация перетаскивания) |
| **Описание** | Система должна позволять пользователю изменять статус задачи на один из трёх предопределённых: «К выполнению», «В работе», «Выполнено». Изменение возможно двумя способами: перетаскиванием карточки в другую колонку или нажатием кнопки смены статуса на карточке. |
| **Входные данные** | Действие пользователя: перетаскивание карточки в целевую колонку или нажатие кнопки смены статуса. |
| **Ожидаемый результат** | 1. Система определяет новый статус на основе целевой колонки (при перетаскивании) или выбранного значения (при нажатии кнопки). <br>2. Система проверяет, что новый статус является одним из трёх допустимых. <br>3. Статус задачи обновляется в локальном хранилище. <br>4. Карточка задачи плавно (с анимацией) перемещается в соответствующую колонку на доске. <br>5. Цвет метки статуса на карточке обновляется в соответствии с новым статусом (синий, жёлтый, зелёный). |
| **Примечания по безопасности** | **Защита от манипуляций:** Изменение статуса должно быть возможно только через UI-элементы (перетаскивание или кнопки). Система не должна обрабатывать прямые изменения статуса через URL-параметры, консоль браузера или другие неавторизованные каналы. <br>**Валидация статуса:** Система должна строго проверять, что новый статус входит в предопределённый список. Любое другое значение должно игнорироваться. |

#### ФТ-03: Удаление задачи

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-03 |
| **Источник** | БТ-04, БТ-07, UC-03, Features (подтверждение через модальное окно) |
| **Описание** | Система должна позволять пользователю удалить задачу, но только после явного подтверждения через модальное окно. |
| **Входные данные** | Действие пользователя: нажатие кнопки «Удалить» на карточке задачи, затем нажатие кнопки «Удалить» в модальном окне подтверждения. |
| **Ожидаемый результат** | 1. После нажатия кнопки «Удалить» на карточке, система отображает модальное окно с текстом «Вы уверены, что хотите удалить задачу?» и двумя кнопками: «Удалить» и «Отмена». <br>2. Если пользователь нажимает «Удалить», задача удаляется из локального хранилища, и её карточка исчезает с доски. <br>3. Если пользователь нажимает «Отмена» или закрывает модальное окно, задача остаётся без изменений. |
| **Примечания по безопасности** | **Защита от случайного/злонамеренного удаления:** Удаление задачи не должно быть возможным без явного подтверждения. Модальное окно должно быть реализовано как отдельный элемент DOM, а не через `alert()` или `confirm()`, чтобы предотвратить возможные атаки, основанные на перехвате этих функций. <br>**Логирование (опционально):** Для аудита можно добавить необязательное требование логировать факт удаления задачи (ID задачи, время) в консоль или локальное хранилище. |

#### ФТ-04: Фильтрация задач по статусу

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-04 |
| **Источник** | БТ-05, БТ-07, UC-04, Features (выпадающий список над колонками) |
| **Описание** | Система должна предоставлять пользователю возможность фильтровать отображаемые задачи по их статусу с помощью выпадающего списка. |
| **Входные данные** | Выбор пользователя из выпадающего списка: «Все», «К выполнению», «В работе», «Выполнено». |
| **Ожидаемый результат** | 1. Система отображает выпадающий список над колонками задач. <br>2. При выборе конкретного статуса (например, «В работе»), система скрывает все задачи, статус которых не соответствует выбранному. <br>3. При выборе «Все» отображаются все задачи без фильтрации. <br>4. Фильтр применяется только к отображению; данные задач в локальном хранилище не изменяются. |
| **Примечания по безопасности** | **Валидация входных данных:** Выпадающий список должен содержать только предопределённые значения. Система не должна принимать произвольные строки от пользователя для фильтрации. <br>**Фильтрация на стороне клиента:** Фильтрация выполняется на стороне клиента (JavaScript), что исключает необходимость в серверной валидации, но требует, чтобы код фильтрации был защищён от модификации через консоль браузера (например, с помощью замыканий или модулей). |

#### ФТ-05: Сохранение данных в локальном хранилище

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-05 |
| **Источник** | БТ-06, Features (автоматическое сохранение при каждом изменении) |
| **Описание** | Система должна автоматически сохранять все данные задач (ID, название, описание, статус) в `localStorage` браузера при каждом их изменении (создание, изменение статуса, редактирование, удаление). При загрузке страницы система должна восстанавливать данные из `localStorage`. |
| **Входные данные** | Данные задач (объекты JSON). |
| **Ожидаемый результат** | 1. При любом изменении данных (создание, обновление, удаление задачи) система сериализует весь массив задач в JSON и сохраняет его в `localStorage` под ключом `myTasker_tasks`. <br>2. При загрузке страницы система проверяет наличие данных в `localStorage`. Если данные есть, они десериализуются и отображаются на доске. Если данных нет, отображается пустая доска. <br>3. Система корректно обрабатывает ситуацию, когда `localStorage` недоступен (например, в режиме инкогнито) или переполнен, выводя соответствующее сообщение пользователю (например, «Не удалось сохранить данные. Проверьте настройки браузера.»). |
| **Примечания по безопасности** | **Защита чувствительных данных:** В данной системе задачи не содержат чувствительной информации (паролей, токенов). Однако, если в будущем планируется добавление таких данных, они должны храниться в зашифрованном виде. <br>**Валидация при загрузке:** При загрузке данных из `localStorage` система должна проверять, что данные являются валидным JSON-массивом объектов с ожидаемой структурой (ID, название, описание, статус). Некорректные данные должны быть проигнорированы, а пользователь должен быть уведомлен. <br>**Очистка хранилища:** При удалении задачи система должна удалять соответствующий объект из массива и перезаписывать `localStorage`. |

#### ФТ-06: Редактирование задачи

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-06 |
| **Источник** | Features (редактирование по двойному клику) |
| **Описание** | Система должна позволять пользователю редактировать название и описание существующей задачи по двойному клику на её карточке. |
| **Входные данные** | Пользовательский ввод: новое `название` (строка, обязательное), новое `описание` (строка, опциональное). |
| **Ожидаемый результат** | 1. При двойном клике на карточке задачи, поля «Название» и «Описание» становятся редактируемыми (например, превращаются в текстовые поля ввода). <br>2. Пользователь может изменить текст и нажать Enter (или кликнуть вне поля) для сохранения. <br>3. Система валидирует новое название: если оно пустое или состоит только из пробелов, отображается сообщение «Название задачи обязательно», и редактирование отменяется (возвращается предыдущее значение). <br>4. Система обновляет данные задачи в локальном хранилище и обновляет отображение на доске. |
| **Примечания по безопасности** | **Валидация и санитизация ввода:** Те же правила, что и для ФТ-01. Все вводимые данные должны быть проверены на длину и очищены от вредоносного кода перед сохранением и отображением. <br>**Кодирование вывода:** При отображении отредактированных данных на карточке, они должны быть закодированы для HTML. |

#### ФТ-07: Отображение счётчика задач

| Атрибут | Значение |
|---|---|
| **ID** | ФТ-07 |
| **Источник** | Features (счётчик задач) |
| **Описание** | Система должна отображать общее количество задач и количество выполненных задач (со статусом «Выполнено») в интерфейсе. |
| **Входные данные** | Данные задач из локального хранилища. |
| **Ожидаемый результат** | 1. В интерфейсе (например, в верхней части доски) отображается текст: «Всего задач: [число], Выполнено: [число]». <br>2. Счётчики автоматически обновляются при любом изменении данных (создание, удаление, изменение статуса задачи). |
| **Примечания по безопасности** | **Только для чтения:** Счётчики являются исключительно отображаемыми элементами и не принимают пользовательский ввод. Уязвимости, связанные с вводом, отсутствуют. <br>**Кодирование вывода:** Числовые значения счётчиков должны быть вставлены в DOM с использованием безопасных методов (например, `textContent`). |
